# 记忆设计
之前的记忆设计比较简陋，有内存加载和文件加载两种方式。在了解了skills 的概念之后，发现这种概念可以很好的迁移到记忆模块上面来。做一个日记实现的可视化记忆。
用一个  memory.md 记录当前的记忆，和一些关于记忆的描述，然后具体需要什么记忆了再去对应目录下面寻找。
然后开始调研发现，已经有了很多现成的实现方案了，比如  MemoU，OpenClaw 的记忆机制也是类似的思想，它分为了 工作区记忆、短期记忆（ai写日记）、长期记忆（jsonl记录所有对话）。
并在此基础上做了很多工程化实现，比如 rag 存储，心跳机制定时重写记忆之类的。

那我就用 java 做一个 MVP 版本试一下。

## 总体设计
openclaw 中的记忆分为三个部分：发给模型的工作记忆，存储所有记忆的短期记忆（jsonl存所有，md存提炼内容），存储用户偏好之类的长期行为的长期记忆。

参照这个设计，同样的方式将记忆分为三个部分：发给模型的工作记忆；存储所有记忆的日记 diary；存储用户片偏好和日记地图的 memory.md。

这个设计里面只有 memory.md 没有 soul.md，系统提示词就放在memory里单独当一个小节吧，记忆就是灵魂，不用两个文件。但这样会有一个问题，必须通过格式严格控制不让 ai 随便改soul部分的提示词。
或者这个应该也要可以改？？还是选择不改吧。暂时直接提示词做，后面用代码做也不难。


修改点：
一点点修改是去掉jsonl存储所有对话，所有对话都会存储在数据库中，如果用户真的需要查找，就调用工具去查找就是，没必要存在记忆里面。
改成使用记日志的形式记录备忘录，至于普通的对话，就按照模型的上下文长度，放入合适的长度的对话，暂定设置为 30吧，我觉得这个短期的上下文不用设计的太长，因为大部分人的短期记忆可能不如模型的上下文。
人也只能想到一个模糊的点，然后去查消息列表，而消息要在前端展示出来就需要存储在数据库中，如果真的用户问道了细节，就再去调用 tool 查找吧，这也是一中模拟人的行为。
而日记的设计，直接采用 YYYY-MM-DD.md的格式吧。

写日记条件有三个，满足其一就写
1）当工作区对话数量达到 40的时候将里面所有内容写一次日记，然后清空工作区内容（这里做法有点暴力了），按照日期追加到今天对应日记的后面，然后同时写 memory.md
2）每隔半个小时，会梳理今天的日志，整理的更符合逻辑。同时整理 memory.md 中的日记地图。
3）将写日记做成一个工具调用，给LLM，当用户让你整理记忆的时候，就开始写日记。

读日记：
1）每次对话都会有传入提炼这两天的日记的提要作为短期记忆，这个提炼内容，会在写日记之后进行刷新。

## 具体实现
使用spring ai 框架，兼容当前的项目代码。

以下是具体的代码实现方案规划（暂不包含底层详细代码，只看结构）：

1. 记忆载体层：VisualizedMemoryManager
   你需要实现一个统一的记忆管理器组件，负责底层文件的读写。注意，我们抛弃 workspace 这个概念，统一称为可视化记忆模块。

职责：

读取/更新 memory.md（严格区分并保护 System Prompt / Soul 区域，只允许修改 User Preference 和 Diary Map 区域）。

按日期读写 YYYY-MM-DD.md 日记文件。

提供获取“近两日日记提要”的方法。

管理当前的短期对话列表（比如维持在 30 条以内的 List<Message>）。

2. 读写拦截层：VisualizedMemoryAdvisor (核心 Advisor)
   这是将你的记忆系统挂载到大模型对话流中的关键桥梁。你将用它替换掉现有的 MessageChatMemoryAdvisor。

请求前 (Advise Request)：

每次用户发消息来，Advisor 从 VisualizedMemoryManager 获取三样东西：

memory.md 里的灵魂提示词和用户偏好。

近两日的日记提要。

当前的短期对话上下文（30 条内）。

将 1 和 2 拼接到 System Prompt 中，将 3 作为历史消息发给模型。

请求后 (Advise Response)：

模型回复后，Advisor 将“User 消息”和“AI 回复”存入短期对话列表。

触发条件 1 判断：检查当前对话列表数量是否达到阈值（如 30 或 40）。如果达到，触发一个异步事件（或调用 Manager），将这批对话送给大模型进行“日记总结”，追加写入今天的 YYYY-MM-DD.md，随后清空短期对话列表。

3. 定时心跳层：MemoryHeartbeatTask (Spring @Scheduled)
   你的设计中提到“每隔半个小时，梳理今天的日志和地图”。这种定时任务无法由用户触发的对话（Advisor）来完成，必须借助 Spring Boot 的定时任务机制。

职责：

使用 @Scheduled(fixedDelay = 1800000) 每半小时唤醒一次。

读取今天的 YYYY-MM-DD.md 内容。

使用一个后台专用的 ChatClient（不需要挂载太多记忆），让大模型对今天的日记进行逻辑重写和润色。

提取关键信息，更新 memory.md 中的 Diary Map（日记地图）。

4. 动作扩展层：VisualizedMemoryTools (Spring AI Tools)
   将大模型主动操作记忆的能力封装为工具。

职责：提供带 @Tool 注解的方法。

forceWriteDiary()：当用户说“帮我整理一下刚才聊的”，大模型可以主动调用此工具，提前触发写日记和清空上下文的流程。

searchChatHistory(keyword)：当用户询问非常久远的细节（不在日记提要中）时，大模型调用此工具去数据库或历史消息表中查询。

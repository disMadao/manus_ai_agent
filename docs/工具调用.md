# 两种方式的实现原理对比

[toc]

基于spring ai版本 

spring ai实现工具调用有两种方式：

通过注解的 Methods as Tools：

```java
class DateTimeTools {

    @Tool(description = "Get the current date and time in the user's timezone")
    String getCurrentDateTime() {
        return LocalDateTime.now().atZone(LocaleContextHolder.getTimeZone().toZoneId()).toString();
    }

}
```

通过bean实现的 function as tools。关键区别就是前者用反射，后者不用。

## 一句话回答

**对于绝大多数场景，[@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) (Method as Tool) 完全够用，Function as Tool 是历史遗留 + 少数特殊场景的补充。比如按名字动态解析 Method 不能开箱即用，Function注册bean后，只需在请求里只写工具名，比如 `.toolNames("weatherFunction")`。**

### 具体场景：按名字动态解析工具

假设调用端只想写 `.toolNames("currentWeather")`，不想持有工具对象引用。

------

**方式一：Function Bean（开箱即用）**

**零额外配置**，`SpringBeanToolCallbackResolver` 会自动从容器里找到这个 Function Bean。

------

**方式二：@Tool 方法（需要额外注册一层）**

**问题来了**：你不能直接 `.toolNames("currentWeather")`，因为默认的 resolver 不认识 `@Tool` 方法。

**核心思路**：启动时把 `@Tool` 方法扫描出来，注册为 `ToolCallback`，塞进 `StaticToolCallbackResolver`。

------

**对比**

|            | Function Bean                     | @Tool + 自定义 Resolver                |
| ---------- | --------------------------------- | -------------------------------------- |
| 调用端代码 | `.toolNames("currentWeather")`    | `.toolNames("currentWeather")`         |
| 工具定义   | 需要 Request POJO + Function Bean | 直接方法 + 注解                        |
| 额外配置   | **无**                            | 需要写 `ToolCallbackResolver` 注册逻辑 |
| 按名字解析 | **开箱即用**                      | 手动桥接                               |

------

**一句话总结**：`@Tool` 方法本身完全能实现按名字解析，但框架没帮你接好这一步，需要你自己把 `ToolCallbacks.from(bean)` 的结果注册进 resolver。Function Bean 之所以"常用于按名字解析"，纯粹是因为默认的 `SpringBeanToolCallbackResolver` 帮你做了这件事。

------

### 两者的本质差异

|                 | Method as Tool                                               | Function as Tool                                             |
| --------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **输入模型**    | 方法参数可以是**多个独立值**、基本类型、List、Map、数组等    | **必须是单一 POJO 对象**，不支持基本类型和集合               |
| **定义方式**    | 直接在方法上写逻辑                                           | 必须实现 [Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 接口或 lambda |
| **Schema 生成** | 从方法签名自动生成                                           | 从泛型参数类型 `I` 的字段生成                                |
| **注册方式**    | 传对象实例 [.tools(new XxxTools())](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) | 传名称 [.toolNames("beanName")](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 懒解析 |

文档中明确列出的**Function Tool 不支持的类型**：

> Primitive types, Optional, Collection types (List, Map, Array, Set), async types, reactive types

而 Method as Tool **全都支持**（除了 Optional、async、reactive）。

这意味着这种简单的工具用 Function 方式反而**更麻烦**：

------

### Function as Tool 存在的真正原因

#### 1. 历史原因（主要原因）

Spring AI 最早（1.0 之前）**只有** Function 方式，沿用了 Spring Cloud Function 的设计哲学——Java 里 [Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 是标准的函数式接口，Bean 化后天然和 Spring 生态兼容。[@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 注解是**后来加的**，是对 OpenAI/LangChain 等框架"tool"概念的对齐。

从代码中能看到痕迹——[MethodToolCallbackProvider](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 里[MethodToolCallbackProvider.java:108](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html)：

两种机制被**强制分离**，说明这是两条独立演化的路径，不是统一设计。

#### 2. 动态 Bean 解析（唯一真正有价值的场景）

Function as Tool 有一个 Method as Tool 做不到的事：**按名称懒加载**。

这在以下场景有意义：

- 工具定义和使用方分离（比如工具在一个 starter 里，使用者只知道名字）
- 工具需要 Spring 管理生命周期（依赖注入、AOP 代理等）
- 按条件动态注册/替换工具

但说实话，[@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 方式的类如果本身是 Spring Bean（`@Component`），也能享受 Spring 管理，只是注册方式不同。

#### 3. 与已有 Function Bean 的兼容

如果你的项目里已经有大量 [Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) Bean（比如从 Spring Cloud Function 迁移过来），不想重写成 [@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html)，那 Function as Tool 让你零改动即可用。

------

### 我的判断：设计不够统一，但可以理解

**这确实是演进过程中的历史包袱，而不是精心设计的两套并行方案。** 证据：

1. Function as Tool 的类型限制（不支持基本类型、集合）比 Method as Tool 严格得多——如果是统一设计，不会有这种不对称
2. 文档里 Function Tool Limitations 章节末尾直接说："Primitive types and collections are supported using the method-based tool specification approach"——等于承认 Function 方式是受限的
3. [@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 注解功能更完善（[returnDirect](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html)、[resultConverter](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html)、[@ToolParam](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 全支持），Function 方式要实现同样效果需要更多手动配置

------

### 实际建议

**新项目全部用 [@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 就对了。** 理由：

1. 更简洁，不需要额外的 Request/Response 类
2. 支持的类型更多
3. [@ToolParam](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 能精确控制每个参数的描述和 required 状态
4. 一个类里可以定义多个工具方法，组织更清晰
5. 和 AI 框架生态（OpenAI、LangChain 等）的"tool"概念对齐

只有两种情况考虑 Function as Tool：

- 你继承了已有的 [Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) Bean，不想重写
- 你需要**纯按名称**动态解析工具，且不想在调用处持有工具类的引用

### 为什么要有两种方式？作者的设计考量

**1) 历史演进：Function 方式是 Spring AI 早期的设计**

Spring AI 最早沿用了 Spring Cloud Function 的思路——在 Spring 生态中，[Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) Bean 是一等公民。这种模式对 Spring 老用户非常熟悉：定义一个 `@Bean`，注册就完事了。它和 Spring Cloud Stream、Spring Cloud Function 的编程模型一脉相承。

**2) [@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 是更新、更直观的 API**

[@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 注解方式是后来引入的（类似于 LangChain/OpenAI SDK 的工具定义风格），优势：

- **更简洁**：不需要创建专门的 Request/Response 类，方法直接带参数就行
- **多参数**：[@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 方法可以有多个参数，会自动生成 JSON Schema；而 [Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 必须封装成单个对象
- **就地定义**：描述、名称、参数说明都在方法上通过注解声明，不需要分散到 Bean 定义和类注解上
- **更灵活**：支持 [@ToolParam](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 标注参数描述和是否必填

**3) 面向不同使用场景**

从代码中的 [MethodToolCallbackProvider.isFunctionalType()](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 可以看到，如果一个 [@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 方法返回 [Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html)/[Supplier](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html)/[Consumer](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html)，框架会**跳过它并警告**——两种机制被刻意分离，避免混淆。这说明作者明确将它们视为两条独立的路径，保持各自的语义清晰。

**总结**：两种方式底层殊途同归——都是构建 [ToolCallback](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 给 LLM 使用。[Function](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) Bean 是 Spring-native 风格的早期方案，[@Tool](vscode-file://vscode-app/d:/sofrware/Microsoft VS Code/072586267e/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 是更现代、更简洁的声明式 API。作者保留两种方式，既保证了向后兼容，也为不同偏好的开发者提供了选择。

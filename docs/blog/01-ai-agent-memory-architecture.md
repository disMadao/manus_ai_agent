# AI Agent 记忆架构演进：从无脑追加到“三层记忆”模型

在构建具有长周期连续工作能力的 AI Agent 时，随着对话轮次的增加，如何有效管理上下文成为了核心难题。本文总结了我们在项目中从“无脑追加”演进到“短期+中期地图+长期RAG”三层记忆模型的实践。

## 痛点：Token 消耗巨大与 LLM 幻觉

在最初的设计中，我们简单粗暴地将用户的每一次对话追加到 `memory.md` 文件中，然后在每次对话时全量输入给 Agent。这很快暴露出了严重的问题：
1. **Token 污染与浪费**：旧对话的细枝末节占用了大量 Token，不仅昂贵，还会导致 Agent 关注点偏移（Lost in the middle）。
2. **日期幻觉**：由于我们依赖 LLM 去更新带日期的 Markdown 文件，模型常常会“脑补”或错乱当天的日期。
3. **短期上下文污染**：每次记忆塌缩（Collapse）时，并没有清空短期对话历史，导致相同内容被反复学习。

## 解决方案：“三层记忆”模式设计

为了解决上述问题，我们将记忆重构为三种模式（层级），实现精准的上下文加载：

### 1. 短期记忆（Short-term Memory）
代表当前正在进行的会话上下文。在触发“记忆塌缩（Memory Collapse）”机制时，我们对这些对话进行总结压缩，随后**彻底清空**这些短期记录。这样既保留了核心结论，又避免了无用 Token 在后续对话中的累积。

### 2. 中期记忆（Mid-term Memory：日记与记忆地图）
中期记忆由两部分构成，旨在让大模型既能聚焦当下，又能感知近期历史，但不过度消耗 Token：
* **今日日记动态注入**：不再通过 LLM 将当天的日记写入常驻系统提示词，而是通过后端（Java `DateUtil`）硬编码读取今日日记文件，**动态拼接**到系统提示词中。这杜绝了 LLM 伪造日期的问题。
* **五日记忆地图（Memory Map）**：将过去 5 天的日记内容提炼为一句话浓缩地图（例如：`[2026-03-15]: 修复了前端 Shift+Enter Bug`），存入 `memory.md`。Agent 可以借此“感知”近期发生过什么，只有当用户问及相关内容时，才去按需读取当天的详细文件。

### 3. 长期记忆（Long-term Memory：RAG 与向量数据库）
代表昨天及更早的日记以及外部的全局知识文档。
由于今天的日记已经全量提供给了 Agent，我们将“今天的日记”排除在 RAG（检索增强生成）系统之外。
每次塌缩时，系统会检查昨天的日记是否已经向量化（Embedded），如果没有则加入 PgVector 向量数据库中，供长期语义检索使用。

## 技术亮点：ChatClient 的环境隔离
在实现记忆塌缩时，我们发现如果复用主对话的 `ChatClient`，大模型输出的压缩结果会被当做一次普通对话继续堆积在会话历史中。为此，我们**隔离出一个专属的 Summarization ChatClient**，让它在后台默默完成阅读、总结、改写五日记忆地图的工作，而完全不污染面向用户的对话上下文。
